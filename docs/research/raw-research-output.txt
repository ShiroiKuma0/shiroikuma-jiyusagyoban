# OpenTasker — Phase 0 Repository State Assessment

## Executive Summary

OpenTasker v0.2.0 (released May 2026) is a fully open-source FOSS alternative to Tasker for Android. The project is **beta-maturity**, with v0.3.0 currently in development. Current implementation includes ~4,700 LOC of Kotlin with core engine operational, 43 built-in actions, 6 context families, and Room DB persistence.

## Code Footprint & Architecture

- **Language:** Kotlin 2.0 (100% native)
- **LOC estimate:** ~4,700 (core source code, excluding tests/resources)
- **Build system:** Gradle 8.9, AGP 8.7.2
- **Min SDK:** API 26 (Android 8.0)
- **Target SDK:** API 35 (Android 15)

### Architecture Layers

```
┌─ Compose UI (Material 3, AMOLED default)
├─ Engine: AutomationService → ProfileMatcher → TaskRunner → ActionRegistry
├─ Action Library (43 built-ins + plugin SDK)
├─ Storage (Room DAOs + StateFlow live queries)
└─ Platform Services (BroadcastReceiver, AccessibilityService, UsageStats)
```

### Core Components Delivered

1. **AutomationService (foreground):** Runs on app init, loads enabled profiles, subscribes to context streams
2. **ProfileMatcher:** Monitors context events, emits profile state transitions (matches/mismatches)
3. **TaskRunner:** Sequential action execution with variable expansion and error handling
4. **ContextSources (6 families):**
   - Application (foreground app via UsageStatsManager)
   - Time (clock windows: 09:00–17:00)
   - Day (weekly schedule: SMTWTFS)
   - Location (geofence skeleton)
   - State (battery ≥80%, headphones, charging, screen, WiFi)
   - Event (SMS, boot, notifications)
5. **ActionMetadata Registry:** 43 actions with dynamic form generation
6. **Run Log System:** Task execution history with cooldown enforcement

### Version History

- **v0.1.0 (May 2026):** Core data model, Room persistence, Compose scaffold, 43 actions, context sources
- **v0.2.0 (May 2026):** Full UI layer (Profile/Task CRUD, action editor, context picker, database integration)
- **v0.3-dev (current):** Execution engine wired (AutomationService → ProfileMatcher → TaskRunner), run log persistence, cooldown logic

## Current Maturity Assessment

**Status:** Beta (operational engine, active development)

### Fully Implemented ✅
- Core data model (Profile, Context, Task, Action, Scene, Variable)
- Room persistence with StateFlow live updates
- Full CRUD UI (profiles, tasks, scenes, variables)
- Action editor with dynamic form generation (43 actions)
- Context picker UI (all 6 families)
- AutomationService foreground integration
- Profile execution pipeline
- Run log viewer
- Task execution with cooldown + error handling
- AMOLED + Catppuccin Mocha theme + light toggle

### Known Gaps ⏳
- **Tasker import:** `.prj.xml` / `.tsk.xml` parser not implemented
- **Variable system:** Basic `%expansion` only; lacks arrays, math operators, regex, JSON parsing
- **Scene editor:** Skeletal only; no visual builder for UI elements
- **Plugin SDK:** AIDL contract designed (Locale-compatible) but not stabilized
- **Advanced contexts:** Geofence refinement (Google Play Services), NFC, notification listener integration incomplete
- **Scheduled execution:** AlarmManager / WorkManager for background scheduled tasks not yet integrated
- **Accessibility:** TalkBack integration not prioritized
